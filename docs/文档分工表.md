# 文档分工表

这份文档用来把当前 docs 目录里的内容正式分工，避免后续继续出现：

- 同一件事在多个文档里重复写
- 新文档和旧文档职责重叠
- 新同学不知道该先看哪份

原则很简单：

- 每份文档只负责一类问题
- 需要“最新实现”时，先看总入口和索引
- 需要“怎么做”时，看规范文档
- 需要“怎么上线”时，看部署和验收文档

---

## 1. 权威入口分工

### 1.1 项目总入口：`README.md`

职责：

- 说明项目是什么
- 说明当前做到什么程度
- 说明怎么启动
- 说明核心路由、角色、测试、开发约定

不要继续往里塞过细的规范、计划或增长策略。

### 1.2 文档索引：`docs/文档导航.md`

职责：

- 负责把 docs 目录的文档分层
- 告诉新同学该先读什么
- 作为文档目录总入口

它只负责“指路”。

### 1.3 使用说明：`docs/使用说明.md`

职责：

- 面向新同学、运营同学的完整使用说明
- 讲清楚项目怎么用、页面怎么走、后台怎么进、投稿怎么提

适合回答：

- 我怎么启动
- 我怎么登录
- 我怎么进入后台
- 我怎么提交投稿
- 我遇到问题先看哪里

---

## 2. 规范类文档分工

### 2.1 开发规范：`docs/项目规范与复用指南.md`

职责：

- 说明当前项目应该怎么开发
- 说明优先复用什么
- 说明页面、Hook、Service、lib、Query 的边界
- 说明后台交互和复用习惯

适合回答：

- 新功能怎么接
- 能不能复用现有组件
- 页面里哪些逻辑不能继续散落

### 2.2 页面级规范：`docs/页面级UI规范.md`

职责：

- 规定首页、详情页、工作台的页面层级
- 规定入口语气
- 规定按钮层级

适合回答：

- 页面主按钮放哪里
- 入口卡怎么分层
- 详情页右侧要留多少说明

### 2.3 后台动作规范：`docs/后台动作与审计规范.md`

职责：

- 统一后台动作口径
- 统一右侧动作区顺序
- 统一用户管理、日志管理、审计闭环的后台动作语言

适合回答：

- 后台按钮该叫什么
- 危险操作怎么放
- 日志页和用户页动作顺序怎么统一

---

## 3. 执行与验收分工

### 3.1 主执行清单：`docs/后台前台执行计划.md`

职责：

- 记录当前主开发任务拆分
- 记录权限、后台、投稿、前台真实数据的推进顺序
- 作为“怎么开发”的任务地图

不要在这里继续写过多上线前检查项。

### 3.2 上线验收：`docs/最终上线验收清单.md`

职责：

- 发布前逐项检查
- 判断能不能正式上线

适合回答：

- 现在能不能发版
- 哪些项没过就不能上线

### 3.3 上线准备：`docs/上线准备清单.md`

职责：

- 作为更宽泛的发布准备参考
- 记录上线前需要确认的基础条件

建议定位为“上线前准备材料”，不要和最终验收单写成重复清单。

### 3.4 上线后风险：`docs/上线后风险清单.md`

职责：

- 记录不阻塞上线、但建议上线后继续做的项
- 作为后续迭代的风险池

适合回答：

- 这次不做会不会影响上线
- 上线后优先补什么

---

## 4. 产品与增长分工

### 4.1 产品增长路线图：`docs/产品增长与优化路线图.md`

职责：

- 讲产品后续怎么吸引用户
- 讲教程、工具、社区、投稿怎么继续形成闭环
- 讲增长策略、转化路径、内容策略

适合回答：

- 接下来做什么更能吸引用户
- 怎么提高回访和传播

### 4.2 内容工具融合方案：`docs/内容工具融合方案.md`

职责：

- 讲教程内容和工具箱怎么融合
- 讲内容页如何嵌入工具
- 讲中期产品交汇点怎么设计

适合回答：

- 教程和工具怎样互相导流
- ND 计算器和教程详情怎么结合

---

## 5. 运维分工

### 5.1 环境变量说明：`docs/环境变量说明.md`

职责：

- 说明本地 / 生产环境需要哪些配置
- 说明哪些变量和项目有关

### 5.2 部署说明：`docs/部署说明.md`

职责：

- 说明部署、迁移、上线步骤
- 说明生产环境怎么配置

---

## 6. 推荐保留方式

以后维护文档时，建议这样处理：

- 项目总概述写 `README.md`
- 新同学上手写 `docs/使用说明.md`
- 开发规则写 `docs/项目规范与复用指南.md`
- 页面体验写 `docs/页面级UI规范.md`
- 后台动作写 `docs/后台动作与审计规范.md`
- 开发任务写 `docs/后台前台执行计划.md`
- 上线验收写 `docs/最终上线验收清单.md`
- 风险池写 `docs/上线后风险清单.md`
- 增长方向写 `docs/产品增长与优化路线图.md`
- 产品融合方案写 `docs/内容工具融合方案.md`
- 部署配置写 `docs/环境变量说明.md` 和 `docs/部署说明.md`

---

## 7. 该合并、不该合并的判断

### 建议不要合并的

- `README.md` 和 `docs/使用说明.md`
- `docs/项目规范与复用指南.md` 和 `docs/页面级UI规范.md`
- `docs/后台前台执行计划.md` 和 `docs/最终上线验收清单.md`
- `docs/项目优化总览.md` 和 `docs/上线后风险清单.md`

因为它们虽然都和项目有关，但回答的问题不同。

### 建议只做引用关系，不做内容合并的

- `docs/文档导航.md` 引用所有文档
- `README.md` 引用关键文档
- 具体专项文档彼此交叉引用，但不要互相重复正文

---

## 8. 一句话总结

现在这套文档最好的结构是：

- 一个总入口
- 一个索引
- 一份使用说明
- 三份规范
- 一份执行计划
- 一份验收单
- 一份风险池
- 两份产品方向
- 两份运维说明

这样就既能保留完整性，也能避免重复。
