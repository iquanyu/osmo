# 上线准备清单

> 目标：把正式发布前需要确认的事项整理清楚，方便按项检查。

## 一、必须上线

这些是影响“能不能正式上线”的核心项，建议在发布前完成。

### 1. 账号与权限

- 确认 `admin / player` 两种角色的边界稳定
- 确认后台路由全部受 `auth + admin` 保护
- 确认玩家登录禁用、角色切换、密码重置等动作都能正确拦截和记录
- 确认未登录、普通玩家、禁用用户在关键页面的跳转行为一致

### 2. 后台主链路

- 教程管理
  - 列表、筛选、详情、编辑、删除可正常工作
  - 新增 / 编辑页面可正常保存
  - 推荐位、状态、排序等运营信息可稳定更新

- 社区管理
  - 列表、详情、置顶、删除、官方回复可正常工作
  - 社区详情页右侧动作栏与列表页动作区保持一致

- 投稿审核
  - 列表、详情、通过、驳回可正常工作
  - 审核状态、驳回原因、入库结果可正确展示

- 用户管理
  - 能查看用户列表
  - 能切换角色
  - 能重置密码
  - 能禁用 / 恢复登录
  - 能查看只读详情
  - 导出可用

- 日志管理
  - 能查看操作日志
  - 能按用户、动作、时间筛选
  - 能查看详情弹层
  - 能导出 CSV

### 3. 前台核心体验

- 首页主入口明确
  - 教程 / 工具 / 社区 / 投稿分流清晰
- 教程详情页能形成“内容 + 工具”闭环
  - ND 计算器入口、参数复制、相关教程推荐都可用
- 工具页可稳定访问
  - ND 计算器、规格页、配件页都能正常运行
- 社区页可稳定浏览和发帖
- 投稿页可稳定创建、编辑、提交、删除

### 4. 数据与基础设施

- 角色迁移、登录禁用迁移、日志表迁移已执行并验证
- 生产环境 `.env`、APP_KEY、数据库、队列、缓存等配置已准备
- 站点在生产环境可正确生成 Vite 资源
- 已确认构建产物可正常加载，不再出现 manifest 缺失

### 5. 测试与回归

- 关键后台页面有对应 Feature Test
- 权限边界有覆盖
- 审核、删除、导出、登录禁用等危险动作有覆盖
- 至少跑一轮完整构建与相关测试

### 6. 监控与恢复

- 基础错误日志可查
- 关键后台操作可追踪
- 有至少一套恢复演示数据或回滚数据的方式
- 重要导出和操作失败时有明确反馈

## 二、上线后再做

这些不是阻止上线的硬门槛，但会显著提升长期运营效率和产品体验。

### 1. 更细的权限体系

- 把 `admin` 再细分成更明确的运营能力
- 例如：
  - 内容运营
  - 审核运营
  - 用户运营
  - 超级管理员
- 对不同动作做更细粒度的授权

### 2. 更完整的用户运营

- 用户详情页
- 登录历史
- 最近操作记录
- 批量角色调整
- 批量禁用 / 恢复
- 更安全的密码重置流程

### 3. 更强的日志与审计

- 日志对象跳转到对应页面
- 更完整的模块映射
- 导出格式增强
- 日志保留策略和归档
- 更细的审计字段标准

### 4. 后台操作体验再精修

- 进一步统一按钮文案和状态文案
- 全局化危险操作确认
- 更强的空状态、加载态、失败态
- 快捷筛选和收藏筛选

### 5. 前台转化与增长

- 首页转化文案 A/B 调整
- 用户行为埋点
- 教程推荐策略优化
- 社区活跃引导
- 投稿转化漏斗优化

### 6. 运营工具扩展

- 批量导出更多数据
- 更细的筛选面板
- 更完整的统计报表
- 更强的内容审核辅助工具

## 三、建议的上线顺序

1. 先确认权限、迁移、构建、测试全部稳定
2. 再做一次后台主链路验收
3. 然后确认前台首页、教程详情、投稿台的核心闭环
4. 最后再决定哪些增强项放到上线后迭代

## 四、结论

如果只看“能不能上线”，当前项目已经接近可发布状态。  
如果看“能不能稳定运营”，还需要把权限分层、审计深度、用户运营和前台转化再补强一轮。
