# 后台动作与审计规范

> 目的：统一后台动作语言、动作顺序和审计表达。

## 1. 适用范围

- 运营总览
- 教程列表 / 教程新增 / 教程编辑
- 社区列表 / 社区详情
- 投稿列表 / 投稿详情
- 用户管理
- 日志管理

## 2. 后台动作统一规则

### 2.1 动作排序

右侧动作区按这个顺序：

1. 主动作
2. 次动作
3. 继续查看类动作
4. 导出类动作
5. 危险操作

`AdminDetailActionPanel` 用 `priority` 控顺序：

- `priority: 0` 主动作
- `priority: 1` 次动作
- `priority: 2+` 其他动作

### 2.2 文案词表

后台动作尽量统一使用这组词：

- `返回总览`
- `返回教程`
- `查看审核`
- `查看社区`
- `查看日志`
- `查看用户管理`
- `查看关联日志`
- `查看关联对象`
- `导出用户`
- `导出日志`
- `通过审核`
- `驳回审核`
- `置顶帖子`
- `取消置顶`
- `删除帖子`

尽量避免这些口径：

- `去总览`
- `去编辑`
- `去审核`
- `去社区`
- `去通过`
- `去驳回`

原则很简单：

- 返回类动作用“返回”
- 查看类动作用“查看”
- 执行类动作用“通过 / 驳回 / 置顶 / 删除 / 导出”

### 2.3 危险操作

危险操作统一放最后，并且必须满足：

- 走确认弹窗
- 使用明确的破坏性文案
- 按钮用 `destructive` 或等价危险视觉
- 成功 / 失败有明确反馈

## 3. 用户管理闭环

用户管理页的目标是可追踪，不是只改状态。

### 3.1 当前闭环能力

- 查看用户基础信息
- 切换角色
- 重置密码
- 禁用 / 恢复登录
- 查看只读详情
- 查看关联日志
- 导出用户
- 显示最近操作摘要

### 3.2 用户页应该表达什么

用户页要让运营快速回答：

- 这个用户是谁
- 当前能不能登录
- 当前是什么角色
- 最近被做过什么操作
- 这个用户的操作是否会影响后续审计

### 3.3 建议继续补的方向

- 最近操作继续按类型分层展示
- 补登录历史
- 补更明确的备注或原因字段
- 让用户页可以继续跳到投稿 / 社区相关内容

## 4. 日志管理闭环

日志页定位为轻量审计中心。

### 4.1 当前闭环能力

- 按操作、操作人、对象类型、时间筛选
- 日志详情弹层
- 原始上下文查看
- 当前条件导出
- 按模块分组查看
- 从用户页跳入关联日志
- 从日志页跳回关联对象

### 4.2 日志页应该表达什么

日志页要让运营快速回答：

- 谁做了什么
- 影响了哪个对象
- 对象属于哪个模块
- 还能不能继续追下去
- 这条记录能不能导出留存

### 4.3 建议继续补的方向

- 日志对象跳转继续完善为更多业务实体直达
- 日志动作枚举化
- 日志模块字典统一
- 大数据量导出改异步任务
- 日志保留策略与归档策略补齐

## 5. 交互语言统一原则

### 5.1 页面语气

- 页面标题用名词短语：`用户管理`、`日志管理`、`投稿审核`
- 右侧动作用动词短语：`返回总览`、`查看日志`、`导出日志`
- 危险动作要直白：`删除帖子`、`驳回审核`

### 5.2 详情页语言

详情页固定三段：

1. 摘要
2. 上下文
3. 原始数据

这个结构适用于日志详情，也适用于后续的其他审计型详情页。

## 6. 复用建议

新增后台详情页时，默认复用这些模式：

- `AdminDetailActionPanel`
- `AdminStatCard`
- `AdminFilterToolbar`
- `AdminInfoRow`
- `AdminInfoTile`
- `Dialog` 详情弹层

新增页面先做三件事：

1. 先定主动作和次动作
2. 再定筛选区
3. 最后补详情弹层和导出

## 7. 结论

当前后台已经进入可持续运营阶段。后续优先继续固定动作顺序、文案口径、用户闭环和审计闭环。
