# 上线后风险清单

> 这份清单只列“建议上线后继续做”的内容：它们不阻塞当前发布，但会影响后续稳定性、可运营性或扩展能力。

## 1. 权限体系仍偏粗

当前只分了 `admin / player` 两类角色，已经足够支撑当前主流程，但后续如果运营协作继续扩大，会遇到这些风险：

- 运营动作粒度不够细，无法区分“内容运营 / 审核运营 / 用户运营 / 超级管理员”
- 所有后台能力目前主要依赖一个粗粒度的后台门禁
- 一旦多人协作，容易出现“谁都能做、谁都不敢做”的边界不清问题

建议上线后继续做：

- 细化权限策略
- 把高风险动作拆成独立授权
- 给不同运营角色配置不同页面 / 不同按钮能力

## 2. 用户管理还缺更完整的运营闭环

当前用户管理已经能满足基础运维，但如果要做成稳定后台，仍有这些后置项：

- 用户详情页还比较轻量
- 没有登录历史、最近操作历史
- 没有批量角色调整
- 没有批量禁用 / 恢复
- 密码重置还是偏“工具型”，还可以更安全、更规范

建议上线后继续做：

- 用户详情页
- 用户安全状态面板
- 最近操作与登录记录
- 批量操作能力

## 3. 日志管理更像“可用”，还没到“审计中心”

当前日志页已经有筛选、分组、详情和导出，但离真正的审计中心还有差距：

- 关联对象还没有做成可跳转的业务入口
- 日志类型仍然以字符串为主，缺统一的模块字典
- 导出功能已经能用，但还可以做成异步任务，避免后面数据多了卡住页面
- 缺少日志归档和保留策略

建议上线后继续做：

- 日志对象跳转
- 日志分级与模块映射
- 日志归档 / 清理策略
- 大数据量导出任务化

## 4. 前台转化与数据闭环还能继续加强

当前首页、教程、工具、社区、投稿都已经打通，但上线后仍然可以继续优化转化：

- 首页入口可以继续做 A/B 文案实验
- 教程详情页的“内容 + 工具”闭环还可以更强
- 用户从首页到投稿台的转化路径还能继续精简
- 社区和教程的推荐策略还可以更智能

建议上线后继续做：

- 行为埋点
- 内容推荐优化
- 首页入口实验
- 教程详情页转化优化

## 5. 运营数据和报表能力不足

现在后台更偏“操作台”，还不是“报表台”。

后置风险：

- 看得到列表，不一定看得到趋势
- 看得到动作，不一定看得到结果
- 很难快速回答“最近一周哪些模块最活跃”“哪个运营动作最常见”

建议上线后继续做：

- 运营报表
- 模块趋势图
- 审核效率统计
- 用户活跃统计

## 6. 可靠性和恢复能力还可增强

当前已经有基础日志和演示数据恢复，但上线后仍建议继续补：

- 更明确的回滚预案
- 更细的异常告警
- 更稳的导出失败处理
- 更完整的数据备份说明

建议上线后继续做：

- 备份与恢复 SOP
- 告警策略
- 导出失败重试
- 关键操作审计强化

## 7. UI 统一性还可以再精修

用户管理和日志管理已经统一了主风格，但还可以继续收细：

- 少量原生控件目前已经替换成统一组件，但页面信息密度差异仍然存在
- 某些后台页的筛选区和内容区还可以再做一层视觉层级统一
- 大屏和小屏下的动作排列还可以更优雅

建议上线后继续做：

- 统一更多后台筛选组件
- 继续抽公共操作区组件
- 细化响应式布局

## 8. 仍建议补的代码级增强

这些不影响当前发布，但对长期维护有价值：

- 日志动作枚举化
- 用户角色语义再细化
- 后台导出统一成一个服务层
- 更严格的类型化表单和筛选参数
- 更多关键链路测试

## 结论

如果按“能不能上线”来判断，当前项目已经可以进入发布准备。  
如果按“能不能长期稳定运营”来判断，上面这些风险点建议上线后继续逐步补齐。
