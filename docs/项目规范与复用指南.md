# 项目规范与复用指南

这份文档只回答两件事：

1. 这个项目现在应该怎么开发
2. 哪些内容已经沉淀为可复用能力

## 1. 项目当前开发原则

### 1.1 先复用，再新增

新增页面、交互、查询或状态定义时，必须先检查以下位置是否已有能力：

- `resources/js/components`
- `resources/js/hooks`
- `resources/js/lib`
- `resources/js/types`
- `app/Services`
- `app/Queries`

如果已有 70% 以上相似能力，优先扩展原有能力，而不是复制页面或重新写一份逻辑。

### 1.2 页面只负责组装，不负责堆业务细节

当前项目已经逐步形成以下职责边界：

- 页面：负责布局、组装组件、接 props
- Hook：负责筛选、请求调用、交互状态
- lib：负责选项、状态文案、默认值、展示语义
- Service / Query：负责服务端数据组织与查询

禁止把以下内容继续散落在页面里：

- 状态 label / tone / 文案判断
- 列表页默认筛选值
- 快捷筛选规则
- 后台请求 URL 拼接
- 重复的卡片结构

### 1.3 后台统一采用“Dashboard 侧边栏 + 列表/详情分离”模式

后台相关页面必须保持统一：

- 运营后台：`/admin/*`
- 玩家后台：`/contribute`

管理后台列表页优先使用：

- 顶部 `DashboardPageHeader`
- 统计卡 `AdminStatCard`
- 筛选条 `AdminFilterToolbar`
- 空态 `AdminEmptyState`
- 分页 `PaginationLinks`

详情页优先采用：

- 左侧主体内容
- 右侧操作 / 概览侧栏
- 信息区块拆分为多个独立 section

### 1.4 后台交互必须统一

后台交互遵循下面规则：

- 确认操作必须走确认弹窗，不允许继续使用原生 `confirm`
- 异步按钮必须有 loading / disabled 状态
- 危险操作按钮必须明确用 destructive 或红色语义
- 后台请求优先走 `use-admin-*` action hooks
- 不允许在页面中手写一堆路由字符串

### 1.5 状态、筛选、选项必须集中定义

凡是满足以下任意一种情况，都应该放到 `resources/js/lib/*`：

- 多个页面共享
- 后续大概率还会扩展
- 页面中会用到 label / filter / default / badge / 空态文案等映射

典型例子：

- 投稿状态：`submission-status.ts`
- 教程分类 / 难度 / 后台快捷筛选：`tutorial-meta.ts`
- 社区排序：`community-meta.ts`

### 1.6 服务端查询必须走 Query / Service 层

控制器中不要继续堆复杂查询与统计逻辑。

约定如下：

- 列表检索、筛选、排序、分页：优先放 `app/Queries/*`
- 首页聚合、页面数据打包、审核/发布等流程：优先放 `app/Services/*`

控制器应该尽量保持轻：

- 验证输入
- 调用 Query / Service
- 返回 Inertia 页面或重定向

### 1.7 类型要先行

前端新增页面时，优先补 `resources/js/types/*`，不要在页面里临时声明一大块本地 interface。

推荐顺序：

1. 先定义 page props 类型
2. 再定义列表 item / filters / meta 类型
3. 页面直接引用共享类型

## 2. 当前项目建议执行规范

### 2.1 新增后台列表页时

必须按这个顺序检查：

1. 是否已有对应 `use-admin-*-actions.ts`
2. 是否已有对应 `use-admin-*-filters.ts`
3. 是否可直接复用 `AdminFilterToolbar`
4. 是否可直接复用 `AdminListCard / AdminListPrimary`
5. 是否已有对应状态 badge / meta 组件

如果没有，再新增；新增后也要尽量做成可复用组件，而不是只服务一个页面。

### 2.2 新增后台详情页时

优先复用：

- `AdminDetailSection`
- `AdminInfoTile`
- `AdminInfoRow`
- `AdminEmptyState`

详情页不要出现大段重复的：

- `Card + CardHeader + CardTitle + CardContent`
- “无数据”文案块
- 信息行布局

### 2.3 新增后台操作时

优先复用：

- `useConfirmDialog`
- `AdminActionButton`
- `use-admin-router`
- `use-admin-action-feedback`

不要在页面里重复做：

- loading 状态切换
- 成功/失败提示散落处理
- 重复的按钮 loading UI

### 2.4 新增前台/后台公共状态语义时

应该优先放到：

- `resources/js/lib`
- `resources/js/types`

不要直接写在页面里的内容包括：

- `status === 'xxx' ? '文案A' : '文案B'`
- `sort === 'date'`
- `'all' | 'pending' | 'approved'` 这类裸联合类型散落多处

### 2.5 路由调用规范

前端请求后端必须优先使用 Wayfinder 生成的路由函数，以及已有 action hook。

优先级：

1. `@/routes/*` / `@/actions/*`
2. `use-admin-*` hook
3. 极少数情况下再补新的统一 action hook

不建议：

- 页面里直接写 URL 字符串
- 每个页面自己封装一次 router 访问

## 3.2 后台 Hook 能力

### 后台统一请求入口

- `resources/js/hooks/use-admin-router.ts`
  - 后台请求统一 router 包装层

### 后台动作 Hook

- `use-admin-overview-actions.ts`
- `use-admin-tutorial-actions.ts`
- `use-admin-community-actions.ts`
- `use-admin-submission-actions.ts`

用途：

- 统一列表跳转
- 统一 CRUD / 审核 / 轻操作
- 统一 Wayfinder 路由调用

约定：

- 新后台页面若需要请求后端，优先补到对应 action hook
- 不建议页面绕过 action hook 直接调用 router

### 后台筛选 Hook

- `use-admin-tutorial-filters.ts`
- `use-admin-community-filters.ts`
- `use-admin-submission-filters.ts`

用途：

- 搜索词状态
- 默认筛选值
- 自动应用/防抖
- 重置筛选
- 快捷筛选切换

约定：

- 新列表筛选逻辑优先进入 hook
- 页面只负责渲染筛选控件

### 通用交互 Hook

- `use-confirm-dialog.tsx`
- `use-admin-action-feedback.ts`
- `use-flash-toast.ts`

用途：

- 确认弹窗
- 成功/失败反馈
- flash 通知整合

## 3.4 前端类型层

关键类型位置：

- `resources/js/types/admin.ts`
- `resources/js/types/tutorial.ts`
- `resources/js/types/community.ts`
- `resources/js/types/submission.ts`
- `resources/js/types/tools.ts`

当前已形成的规范：

- 页面 props 有共享类型
- 列表 filters 有共享类型
- 列表 item / 详情 item 有共享类型
- 投稿、教程、社区都已有明确领域类型

约定：

- 新页面不要继续在页面文件里写大段本地 interface
- 若类型可能被第二个页面使用，就要提升到 `resources/js/types/*`

## 4. 新功能开发时的推荐路径

### 4.1 如果是新增后台列表页

推荐顺序：

1. `types` 里补 page props / filters / list item
2. `lib` 里补状态/筛选语义
3. `Query` 里补列表查询
4. `use-admin-*-actions` 补路由动作
5. `use-admin-*-filters` 补筛选状态
6. 页面中优先复用 `AdminFilterToolbar`、`AdminListCard`、`AdminListPrimary`

### 4.2 如果是新增后台详情页

推荐顺序：

1. 先整理 page props 类型
2. 如果有复杂聚合数据，先补 Service
3. 页面区块优先用 `AdminDetailSection`
4. 信息展示优先用 `AdminInfoTile` / `AdminInfoRow`
5. 交互按钮优先用 `AdminActionButton`

### 4.3 如果是新增审核/危险操作

推荐顺序：

1. 先看是否应进入 `SubmissionService` 或对应 Service
2. 前端操作走 `useConfirmDialog`
3. 请求走 `use-admin-*` action hook
4. 反馈走统一 success / error 模式

## 5. 当前明确不建议的做法

以下做法在当前项目里应尽量避免：

- 在页面里硬编码状态映射文案
- 在页面里手写多个 `router.get/post/patch/delete`
- 列表页自己维护一套筛选默认值和重置逻辑
- 重复写 `CardHeader / CardTitle / CardContent`
- 复制一个后台页面再改字段
- 使用原生 `confirm`
- 用本地 interface 替代共享类型
- 在控制器中堆复杂查询和业务流程

---

## 6. 当前最值得继续沉淀的复用方向

如果后续继续整理，优先级建议如下：

### 高优先级

1. 后台详情页右侧侧栏进一步统一
2. 教程管理列表也逐步向统一列表模式靠拢
3. 后台成功/失败 toast 与错误反馈再统一一层
4. 表单区块（教程编辑 / 投稿编辑）沉淀更多共享字段组件

### 中优先级

1. 社区详情“回复卡片”抽为组件
2. 审核详情“步骤 / 技巧列表项”抽为组件
3. 后台页面级 loading / skeleton 统一

### 低优先级

1. 更进一步的数据 DTO / ViewModel 化
2. 更细颗粒度的权限策略与角色扩展

---

## 7. 给后续开发者的一句话约定

这个项目已经不适合“复制页面快速改一版”的开发方式了。

后续新增内容时，默认策略应该是：

- 先找有没有现成的组件、Hook、lib、Query、Service
- 能扩展就扩展
- 实在没有，再新增
- 新增后也尽量让它成为下一次还能复用的能力

如果大家都按这个约定走，项目会越来越稳；如果继续回到“页面里直接堆逻辑”，后台很快又会重新变散。
